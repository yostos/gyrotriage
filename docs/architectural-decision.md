# ADR-001: 実装言語としてRustを採用する

## ステータス

承認済み

## コンテキスト

gyrotriageは、DJI FPVドローンが撮影したMP4ファイルからモーションデータ（クォータニオン）を抽出・解析し、映像のブレ度合いをスコアリングするCLIツールである。

実装言語の選定にあたり、以下の要件がある。

- DJI独自のprotobuf形式で格納されたモーションデータを正確にパースできること
- 大容量のMP4ファイル（数GB）をバッチ処理できるパフォーマンス
- CLIツールとしてシングルバイナリで配布できること
- クォータニオン演算（角変位量算出、3軸分解）を高速に処理できること

## 検討した選択肢

### Rust

- Gyroflow自体がRustで書かれており、コアライブラリである telemetry-parser がDJI MP4からのクォータニオン抽出を実装済み
- telemetry-parserはDJI独自のprotobuf形式（dvtm_dji_neo.proto等）の解析を含む、検証済みの実装を提供する
- シングルバイナリにコンパイルでき、ランタイム依存なし
- メモリ安全性と高パフォーマンスを両立

### Python

- 開発速度が速く、プロトタイピングに適する
- MP4パース（pymediainfo等）やクォータニオン演算（scipy, numpy）のライブラリが豊富
- ただしDJI独自のprotobuf形式に対応した既存ライブラリがなく、テレメトリ解析部分を自前実装する必要がある
- 大容量ファイルのバッチ処理ではパフォーマンスが課題になりうる
- 配布にはPython環境が必要

### Go

- シングルバイナリ配布が容易
- CLIツール開発の実績が豊富
- ただしDJIテレメトリ解析の既存ライブラリがなく、telemetry-parserに相当する部分を自前移植する必要がある

## 決定

**Rustを採用する。**

## 根拠

最大の理由は、Gyroflowプロジェクトの **telemetry-parser** クレートをそのまま利用できることである。

DJIのモーションデータはMP4内にprotobuf形式で埋め込まれており、そのスキーマは機種・ファームウェアごとに異なる（例: Neo用の `dvtm_dji_neo.proto`）。このパース処理を一から実装するのは労力が大きく、機種追加やファームウェア更新への追従も困難になる。telemetry-parserはGyroflowコミュニティによって継続的にメンテナンスされており、DJI Avata / Avata 2 / Neo / Neo2を含む幅広い機種に対応済みである。

他の言語を選択した場合、telemetry-parserの機能を自前で再実装するか、FFIで呼び出す仕組みが必要になり、いずれも開発・保守コストが増大する。

副次的な利点として、シングルバイナリ配布、高いパフォーマンス、メモリ安全性もCLIツールの要件に合致する。

## 影響

- telemetry-parser クレートへの依存が発生する。Gyroflow本体のバージョンアップに追従する必要がある
- Rustの学習コストがあるが、CLIツールとしてのスコープは限定的であり、複雑なアーキテクチャは不要
- plotters（グラフ生成）、clap（CLI引数パース）など、Rustエコシステムの成熟したクレートを活用できる
