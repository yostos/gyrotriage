# gyrotriage 構想書

## 背景と動機

DJI AvataやNeoなどのFPVドローンで撮影した映像は、飛行状況によってブレの程度が大きく異なる。Gyroflowによる後処理スタビライゼーションは非常に有効だが、すべてのファイルに適用するのは時間がかかる。撮影後に大量のMP4ファイルから「補正が必要な映像」を素早く選別できるツールがあれば、ポストプロダクションのワークフローを効率化できる。

## コンセプト

DJI Avata / Neo が MP4ファイル内に記録するモーションデータ（クォータニオン）を抽出・解析し、映像のブレ度合いをスコアリングして、Gyroflowによるスタビライゼーションの要否を判定するCLIツール。

## ツールの位置づけ

本ツールは、Gyroflowによるスタビライズ作業の「前段にある判断支援ツール」である。自動処理パイプラインの一部ではない。

想定するワークフローは以下の通り。

1. 本ツールで撮影素材を一括解析し、ブレの度合いを定量的に把握する
2. 解析結果をもとに、人間がGyroflowのパラメータを決定する
3. Gyroflowスタンドアロン（またはCLI）でスタビライズ済み動画を書き出す
4. スタビライズ済み動画をDaVinci Resolveに読み込んで編集する

Gyroflow OpenFXプラグインでResolve内から直接スタビライズすることも可能だが、編集で1本の動画を複数カットに分割して使う場合、カットごとにプラグインを適用する手間が発生する。そのため、編集前にGyroflowスタンドアロンで元動画単位でスタビライズを済ませ、処理済みファイルをResolveに持ち込むほうが効率的である。

### 解析出力が意味するもの

本ツールが出力するブレスコアや統計値は、Gyroflowの最適パラメータそのものではない。これらは素材の状態を定量的に把握するための指標であり、主な用途は以下の通り。

- スタビライズが不要なクリップの仕分け。ブレの少ない素材にGyroflowを適用する無駄を省く
- 作業の見通し。ブレが大きい素材はsmoothness値を上げる必要があるが、それに伴いクロップ量も増える。解析結果から「この素材はクロップがきつくなりそうだ」といった事前の見積もりが可能になる
- 素材間の相対比較。複数ファイルのスコアを比較し、どの素材に重点的に時間をかけるべきかを判断できる

smoothnessなどの最終的なパラメータは、Gyroflowのプレビューで映像を見ながら人間が決定する。本ツールはその判断の出発点を提供するものである。

## 対象ハードウェア

- DJI Avata / Avata 2
- DJI Neo / Neo2

いずれもGyroflowに対応しているが、モーションデータ（クォータニオン）が記録される撮影条件は機種によって異なる。

### Avata / Avata 2 の場合

EIS（Rocksteady）をオフ、FOVをWideに設定して撮影する必要がある。

### Neo / Neo2 の場合

Neoのカメラは固定画角（FOV 117.6°、35mm換算14mm、f/2.8）であり、FOVの選択肢（Wide/Normal/Ultrawide）は存在しない。

EIS（Rocksteady/HorizonBalancing）のオン・オフは、アスペクト比と接続コントローラによって決まる。

- **16:9で撮影した場合**: RocksteadyまたはHorizonBalancingが強制適用され、オフにできない。モーションデータは記録されない
- **4:3で撮影した場合**: EISが自動的にオフになり、モーションデータが記録される

コントローラによる制約:

- DJI Flyアプリや標準送信機では、RocksteadyとHorizonBalancingが常時オンで切れない
- DJI Goggles N3 / RC Motion 3 などの没入型モーションコントロール使用時は、EISのオン・オフを選択できる

したがって、本ツールの対象となるNeo映像の撮影条件は以下の通り。

- **アスペクト比: 4:3**（必須。4K 3840x2880@30fps または 1080p 1440x1080@60/50/30fps）
- **コントローラ: DJI Goggles N3 + RC Motion 3**（推奨。DJI Flyアプリでは4:3でもEIS制御に制約がある可能性がある）

## 技術的基盤

### DJIのモーションデータについて

DJIのドローンはIMUの生データ（ジャイロスコープの角速度）を直接記録せず、機内で算出済みのクォータニオン（カメラの3D姿勢）をMP4のメタデータとして格納する。Gyroflowはこのクォータニオンを Integration method: None で直接利用してスタビライゼーションを行う。

本ツールでは同じクォータニオンデータを抽出し、フレーム間の姿勢変化量を数値化することで、ブレの程度を定量評価する。

### 解析手法

1. MP4からクォータニオンの時系列データを抽出する
2. 連続するフレーム間のクォータニオン差分から角変位量を算出する
3. 角変位量をピッチ・ロール・ヨーの3軸に分解する
4. 各軸および合成値の統計量（平均、標準偏差、最大値、ピーク頻度など）からブレスコアを算出する
5. 閾値に基づいてGyroflow適用の要否を判定する

### 判定レベル（仮）

- STABLE … ブレがほぼなく、補正不要
- MILD … 軽微なブレあり、補正は任意
- MODERATE … 明確なブレあり、Gyroflow推奨
- SEVERE … 激しいブレ、Gyroflow強く推奨

閾値は実際の映像とGyroflow適用後の効果を見ながらチューニングする。

## 実装言語

Rust を採用する。Gyroflow自体がRustで書かれており、そのコアライブラリである telemetry-parser がDJI MP4からのクォータニオン抽出を実装済みのため、データ抽出部分の信頼性が高い。

## CLI インターフェース

```
gyrotriage <FILE>            # 単一MP4ファイルの解析
gyrotriage <FILE> --visual   # Sixelグラフ付き出力
gyrotriage <FILE> --html     # HTMLレポート生成
```

単一ファイルのみ受け付ける。複数ファイルの処理はシェルに委ねる（→ ADR-002）。

```bash
for f in *.MP4; do gyrotriage "$f"; done
```

## 出力

### 基本出力（テキスト）

ファイルごとに以下を表示する。

- ファイル名
- 総合ブレスコア（0〜100）
- 判定結果（STABLE / MILD / MODERATE / SEVERE）
- 各軸（ピッチ・ロール・ヨー）の統計サマリー
- モーションデータの有無チェック（EISオンで撮影されたファイルの検出）

バッチ処理時は一覧表形式でまとめ、スコア順にソートするオプションも設ける。

### グラフィカル出力（--visual）

Wezterm の Sixel / iTerm2 インライン画像プロトコルを利用し、ターミナル上に直接グラフを描画する。

表示内容の候補は以下の通り。

- 時系列ブレ強度グラフ（横軸: 時間、縦軸: 角変位量）。閾値ラインを重ねて問題区間を可視化する
- 3軸分解表示（ピッチ・ロール・ヨーを色分けした折れ線）。どの方向の揺れが支配的かを把握できる
- 区間ヒートマップ（動画を数秒単位で区切り、安定度を色のグラデーションで表現する横バー）

Rustでplottersライブラリなどを用いてPNG画像を生成し、Sixelプロトコルでターミナルに送出する。Sixel非対応環境ではANSIカラーのブロック文字（▁▂▃▄▅▆▇█）によるスパークラインにフォールバックする。

### HTMLレポート（--html）

ブラウザで閲覧可能なレポートファイルを生成する。インタラクティブなグラフ（マウスオーバーで値表示など）を含む。

## 依存関係（想定）

- telemetry-parser（Gyroflowプロジェクト）… DJI MP4からのモーションデータ抽出
- clap … CLIインターフェース
- plotters … グラフ画像生成（v2）
- sixel-rs 等 … Sixelプロトコルでのターミナル画像出力（v2）

## 将来的な拡展の可能性

- SRTファイルとの突合（GPS情報と組み合わせて、ブレの発生地点を地図上にマッピングする）
- 問題区間のタイムコード出力（DaVinci Resolveのマーカーとして読み込める形式でのエクスポート）
- GoPro等の他カメラへの対応拡張

## 課題と懸念事項

- 閾値のチューニングには実際の映像での検証が必要。異なる飛行パターン（ホバリング、前進飛行、アクロバット）で妥当な判定ができるか
- ブレスコアとGyroflowパラメータの対応関係は単純ではない。smoothnessの適切な値はブレの大きさだけでなく、撮影意図（パンを活かしたいか完全に安定させたいか）、許容できるクロップ量、FOVなどに依存する。Gyroflow自体にも自動推奨機能がないのはこの困難さを反映している
- telemetry-parser のAPIが安定しているか、Gyroflow本体のバージョンアップに追従する必要があるか
- モーションデータの記録レートが機種やファームウェアによって異なる可能性
- クォータニオンからの角変位量算出において、意図的なカメラワーク（パン・チルト）とブレをどう区別するか
